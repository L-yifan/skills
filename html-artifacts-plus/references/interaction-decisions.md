# 交互决策指南

不是所有 HTML 制品都需要交互。交互的目的是让用户**操作内容**，不是为了"看起来更高级"。添加不必要的交互会分散注意力、增大文件体积、增加 bug 可能。

## 核心原则

**交互服务于操作，不服务于装饰。**

## 三种制品模式与交互需求

| 制品模式 | 用户行为 | 需要的交互 | 不需要的交互 |
|---|---|---|---|
| **阅读型** | 看、对比、理解 | 折叠/展开、tab 切换、跳转锚点 | 拖拽、编辑、排序、表单 |
| **投屏型** | 讲、翻页 | ← → 翻页、F 全屏、页码指示器 | 滚动、编辑、筛选 |
| **操作型** | 改、排序、筛选、导出 | 拖拽、表单、切换、滑块 + **导出按钮** | — |

## 具体交互的适用条件

### Tabs（标签页切换）

**何时用**：
- 同一话题有多个平行视角（如"TypeScript 实现 vs Python 实现 vs 测试用例"）
- 代码示例需要切换语言/框架
- 内容量大致相同，结构平行

**何时不用**：
- 内容有先后顺序（应该用滚动或折叠）
- 只有 2 个 tab 且内容很短（直接用标题分段）
- 移动端 tab 过多（超过 4 个 tab 在窄屏上难以点击）

### Collapsible sections（折叠区）

**何时用**：
- 有深度细节但不是所有读者都需要（如"实现细节""FAQ""附录"）
- 事故报告中的"时间线详情"
- 代码讲解中的"源码展开"

**何时不用**：
- 核心内容（读者必须看的不能折叠）
- 只有 1-2 句话的短内容（折叠反而增加点击成本）

### Drag-and-drop（拖拽排序）

**何时用**：
- 用户需要重新排序/分类/分桶（如分类看板、优先级排序）
- 用户可以直观地判断顺序

**何时不用**：
- 只读内容
- 移动端（拖拽在触屏上体验差，应提供按钮替代方案）
- 项目数少于 3 个（拖拽操作成本高于直接编辑）

### Sorting / Filtering（排序/筛选）

**何时用**：
- 列表超过 ~8 项，且用户需要按不同维度查看
- 数据表有多个可排序列
- 用户需要快速定位特定条目

**何时不用**：
- 列表很短（≤ 8 项，肉眼扫描更快）
- 只读报告中的表格（除非用户明确需要探索数据）

### Sliders / Controls（滑块/控件）

**何时用**：
- 调参场景（动画参数、算法参数、设计 token）
- 需要实时预览的参数变化

**何时不用**：
- 只需要选择固定选项（用 select 或 radio button）
- 阅读型制品中作为装饰

### 导出按钮

**何时必须加**：
- 操作型制品（编辑器、分类板、调参器）——**必须有导出路径**
- 用户需要把操作结果带回工作流中

**何时不加**：
- 阅读型制品（除非用户明确要求复制内容）
- 投屏型制品

**导出格式选择**：
- 编辑器 → Copy Patch JSON + Copy Markdown Diff + Copy Agent Instruction
- 分类板 → Copy as Markdown
- 调参器 → Copy CSS / Copy Config / Copy Code
- 数据表 → Copy CSV / Copy JSON

## 交互密度检查

生成前问自己：

1. **这个制品是给用户读的还是操作的？** 如果只是读，默认不加交互。
2. **每加一个交互，用户的操作路径是什么？** 不要在只读报告上加一个孤立的排序按钮。
3. **移动端能用吗？** hover → tap 替代、拖拽 → 按钮替代、多列 → 堆叠。
4. **交互结果能导出吗？** 如果用户操作了但无法带走结果，交互是死胡同。

## 常见交互滥用

- **在报告上加排序按钮。** 报告是作者编排好的叙事，读者不应该重排。
- **在对比页上加 tabs。** 对比的核心价值是「同时看到」——tabs 正好破坏了这一点。对比应该用列。
- **在移动端做拖拽。** 触屏拖拽体验差。用「上移/下移」按钮或 select 排序替代。
- **折叠核心内容。** "实现方案"是正文不是附录，不应该折叠。
- **只读图表加 hover 工具提示但没有移动端替代。** 移动端没有 hover。关键数据应该直接显示在图表上。
- **加了导出按钮但导出格式不对。** 编辑器导出 JSON 才有意义；报告导出 Markdown 才有意义；不要给编辑器加"Copy CSV"。
